## 4. 上一輪學到的硬規則（**新 Claude 必讀**）

### 4.1 Brief 寫法守則（沿用 + 強化）

- **每份 brief 結尾必須有「禁止觸碰」清單，逐一列出檔案絕對路徑**。W3 三件每份 do-not-touch 清單 25+ 檔，Codex 全守住。
- **強調「不要新增 npm 依賴」如果情境允許**。W3-T03 brief 嚴守 0 新 dep（手寫 30 行 helper），Codex 也照做。
- **不要平行開兩個有檔案重疊的 brief**。
- **完成回報模板要求 Codex 整段貼最敏感檔的 git diff**（如 W3-T03 要求貼 auth.ts 完整 diff）— 強制透明、好 review。
- **不准動 `src/index.ts` 的 path-scoped middleware**（auth 4 行不准合併成 `:id?`）— 上輪在每份 W3 brief 都明確寫了，Codex 都守住。

### 4.2 環境陷阱（沿用）

- **Bash sandbox 與 Read tool 不同步**。`/sessions/.../mnt/Lerna/` 裡 `cat` 出來的內容可能是 stale snapshot，Read tool 才是真實當下狀態。**驗收檔案內容一律用 Read，不用 bash cat**。
- **Bash sandbox 不能刪檔/動 git**。用 PowerShell 內聯指令給使用者跑。
- **PowerShell 是 5.x 不是 7.x**：別用 `pwsh`，用 `powershell` 或內聯指令。
- **PowerShell backtick 多行續行容易踩雷**：W3-T01 push 第一次 commit 因 backtick 沒接好 + lock file 路徑寫錯（`services/api/package-lock.json` 不存在，monorepo lock 在 root）導致 commit 失敗、訊息與內容不一致。**所有 git 指令一律用單行 + root `package-lock.json`**。

### 4.3 驗收紀律（沿用 + 強化）

- 每個 brief 完成後**至少 Read 該 brief 動到的所有檔**。
- 主程式入口檔（如 `src/index.ts`）每輪都要 Read 一次完整內容。
- **完成 + commit + push 才算結案，再開下一件**。W3 期間使用者一度沒 commit W3-T02 就接 W3-T03，導致 working tree 累積兩件變動。Codex 沒亂動其他人的檔但 commit 時要拆兩個比較乾淨——這個流程紀律下次嚴一點。

### 4.5 W3 期間驗證過的 Hono 4 行為（給未來設計參考）

- `c.header('Content-Type', 'application/problem+json')` + `c.json(...)` → c.json **會 overwrite** content-type 回 application/json。**要設自訂 content-type 必須用 `new Response(JSON.stringify(...), { status, headers })`**。`services/api/src/lib/problem.ts` 是參考實作。
- `@hono/zod-openapi` 的 zod schema 必須先 `extendZodWithOpenApi(z)` 才能 `.openapi(...)`。在 schemas 檔 top-level 跑一次即可（idempotent）。
- `OpenAPIHono({ defaultHook })` 的 hook 是把 zod 失敗導向自訂錯誤格式的關鍵；W3-T01 用過、W3-T03 改成 `validationFailed` helper。
- `hono-rate-limiter` 0.5.3 原生支援 `handler` option，可直接用來控 429 的 Content-Type / body shape。
